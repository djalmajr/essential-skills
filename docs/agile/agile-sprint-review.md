# Sprint Review

Prepares review and demo of deliveries for stakeholders.

## When to use

- At the end of a sprint
- To present results to team or stakeholders
- To validate that the product is on the right track
- Collect feedback for the next cycle

## When NOT to use

- Sprint still in progress → wait
- Internal consolidation → `/status-report`
- Formal technical delivery closure → `/post-impl`

## How to use

```
/sprint-review
```

With context (sprint, period):

```
/sprint-review Sprint 23
/sprint-review Q2 2026 - payments
```

## What sprint review includes

1. **Consolidated deliveries** — vs commitment from planning
2. **Demo prepared** — by business value, not by technology
3. **Undelivered items** — with reason and destination
4. **Scope changes** — recorded and explained
5. **Stakeholder feedback** — collected for next cycle

## Tip

Start with **impact**: "now the user can do X" is better than "we implemented feature Y".

## Chaining

- After sprint review → `/retro`
- Next sprint → `/sprint-planning`
